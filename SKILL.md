# gstack Course — Adaptive Learning System

## Self-Update Check

On first use per session, check for updates:

```bash
COURSE_DIR="$(dirname "$(realpath "$0")" 2>/dev/null || echo ~/.openclaw/skills/gstack-course)"
VERSION_FILE="$COURSE_DIR/version.json"
UPSTREAM_SHA_FILE="$COURSE_DIR/references/upstream-sha.txt"

# Show current version
[ -f "$VERSION_FILE" ] && echo "gstack-course $(python3 -c "import json; print('v'+json.load(open('$VERSION_FILE'))['version'])" 2>/dev/null || echo "v?")" || echo "gstack-course v0.1.0"

# Check if GitHub has newer version (non-blocking)
REMOTE_SHA=$(curl -sf "https://api.github.com/repos/OWNER/gstack-course/commits/main" 2>/dev/null | python3 -c "import sys,json; print(json.load(sys.stdin)['sha'])" 2>/dev/null || echo "")
LOCAL_SHA=$(cat "$UPSTREAM_SHA_FILE" 2>/dev/null || echo "")
if [ -n "$REMOTE_SHA" ] && [ "$REMOTE_SHA" != "$LOCAL_SHA" ]; then
  echo "UPDATE AVAILABLE — run: cd $COURSE_DIR && git pull"
fi
```

Replace `OWNER` with the actual GitHub org/user once repo is published.

## Dynamic Skill Loading

**All skill data comes from generated files — never hardcoded.**

1. Read `references/skill-index.json` for structured data (names, phases, sizes, descriptions)
2. Read `references/skills-catalog.md` for human-readable deep reference
3. Read `references/workflow-patterns.md` for sprint patterns and combos

These files are auto-regenerated by GitHub Actions whenever garrytan/gstack changes.

## Progress Tracking

Read `~/.gstack-course/progress.json`. Create if missing:

```json
{
  "version": "0.1.0",
  "last_session": null,
  "skills": {},
  "completed_patterns": []
}
```

Each skill entry: `{ "level": 0-3, "last_taught": "ISO date", "notes": "" }`
- **0** = never seen
- **1** = introduced (knows what it does)
- **2** = practiced (has used or walked through example)
- **3** = mastered (used in real workflow, understands when/why)

**Do not re-teach skills at level 2+ unless the user asks or the skill changed upstream.**

## Teaching Protocol

### "Which skill should I use?"

1. Read `references/skill-index.json` for current skill list
2. Ask what they're trying to do (1 question max)
3. Map to sprint phase: Think → Plan → Build → Review → Test → Ship
4. Recommend 1-2 skills with one-line rationale from index
5. If skill is level 0, give the 3-sentence intro from catalog
6. If level 1+, just say "use /X — you know this one"

### "Teach me /X"

1. Read `references/skill-index.json` to find the skill
2. Check progress level
3. **Level 0→1**: Read `references/skills-catalog.md` for that skill's section. Deliver:
   - What it does (1 sentence)
   - When to use it (1 sentence)
   - Key concept (the non-obvious thing)
   - Example interaction from the catalog
4. **Level 1→2**: Walk through a real scenario from their current project
5. **Level 2→3**: Combine with other skills in a sprint pattern (see workflow-patterns.md)
6. Update `~/.gstack-course/progress.json` after each teaching interaction

### "What's new in gstack?"

1. Read `references/upstream-sha.txt` for last synced commit
2. Check `references/skill-index.json` for `generated_at` timestamp
3. If the repo has updates available, suggest `git pull` in the course dir
4. Report: skill count, any new skills since user's last session, changed skills

### "Show my progress"

Read `~/.gstack-course/progress.json`. Display:
- Skills by level (mastered / practiced / introduced / unseen)
- Suggested next skill to learn
- Workflow patterns completed vs available

## Sprint Workflow — The Core Pattern

gstack is a process, not a toolbox. Skills chain in order:

```
/office-hours → /plan-ceo-review → /plan-eng-review → [build] →
/review → /qa → /ship → /document-release → /retro
```

For full patterns, read `references/workflow-patterns.md`.

## Key Philosophy: Boil the Lake

AI makes completeness nearly free. Always choose "complete" over "good enough."
- A "lake" (full test coverage for a module) = boilable
- An "ocean" (rewriting a dependency) = not boilable
- Show both estimates: human time vs AI+gstack time
